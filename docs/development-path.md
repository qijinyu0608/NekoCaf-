# NekoCafé 开发路径

这份文档用于说明 `foundation -> lab1 -> lab2 -> lab3 -> lab4 -> project/` 的连续开发流程，避免把四次实验做成彼此割裂的独立作业。

## 1. 总体路径

项目按软件工程生命周期推进：

1. `foundation`
   统一维护案例、术语、需求编号、追溯规则和目录约束。
2. `lab1`
   完成需求获取、需求基线、SRS、UML、验证报告和 AI 使用记录。
3. `lab2`
   承接实验一基线，完成架构驱动力分析、DDD 划分、候选架构对比、C4、OpenAPI、ER、ADR 和路线图。
4. `lab3`
   在 `project/` 代码骨架上实现两个核心服务的最小可运行链路，并补容器化、流水线、部署与观测。
5. `lab4`
   在同一工程上补测试体系、质量门禁、缺陷追踪、AI 评审与质量度量。

## 2. 核心原则

- 共享事实源只维护一份，优先以 `docs/foundation/` 为准。
- 实验之间采用同一套编号：`FR / NFR / UC / BC / SVC / TC`。
- 文档、图稿、代码、导出件和提交件分目录管理，不混放。
- 可运行范围优先聚焦 `reservation-service` 和 `member-service`。
- 设计表达范围保留 `order`、`store-ops`、`cat-health`、`recommendation`。
- 所有通过脚本生成的 Word 文档必须统一使用黑色字体，不允许出现主题蓝色标题。
- Git 提交信息采用 `Conventional Commits`，作为跨实验强制性要求，不接受自由格式提交信息。
- 每次实验完成后都要在 Git 仓库打阶段性 tag；实验一固定为 `requirements-v1.0`，后续实验沿用同类版本化命名。

## 2.1 Git 归档强制要求

为保证实验过程可评审、可追溯、可答辩，仓库层必须满足下面规则：

1. 所有正式提交统一使用 `Conventional Commits`
   例如：`feat(lab2): add C4 container view`、`fix(lab1): add requirement testability traceability`、`style(docs): enforce black text policy`
2. 每个实验至少保留一条可识别的阶段完成标记
   实验一：`requirements-v1.0`
   实验二建议：`architecture-v1.0`
   实验三建议：`delivery-v1.0`
   实验四建议：`quality-v1.0`
3. 归档 tag 应对应可复核的提交节点，避免打在临时草稿或未整理版本上
4. 若需要改写不合规的历史提交信息，应在阶段收尾前一次性完成，并保证远端仓库与本地规则一致

## 3. 阶段拆分

### 3.1 Foundation：共享基线

目标：
- 固定案例背景、干系人、目录规范、命名规范和实验之间的映射关系。

主要输入：
- 总任务书
- 各实验任务书

主要产物：
- `docs/foundation/project-overview.docx`
- `docs/foundation/requirements-baseline.xlsx`
- `docs/foundation/glossary-rtm-master.xlsx`
- `docs/foundation/artifact-map.docx`

后续作用：
- 为实验一到实验四提供统一术语、统一需求主表和统一命名规则。

### 3.2 Lab 1：需求与建模基线

目标：
- 把业务问题收敛成后续可以设计、实现、测试的正式输入。

当前状态：
- 已完成第一轮正式交付，形成可复用需求基线。

主要输入：
- `foundation` 中的案例、需求主表和术语主表。

主要产物：
- 需求获取计划与访谈剧本
- 需求清单与 MoSCoW 分级
- SRS
- UML 图集
- 术语表与 RTM
- 需求验证报告
- AI 使用记录
- 答辩 PPT

输出给实验二的关键内容：
- `30+ FR` 与 `10+ NFR`
- `UC` 用例边界
- `BC` 候选业务域线索
- 术语与追溯规则
- 核心业务场景与质量约束

### 3.3 Lab 2：架构与微服务拆分

目标：
- 回答“为什么这样拆、为什么这样设计、如何满足质量属性”的架构问题。

推荐完成顺序：
1. 驱动力分析：从实验一 NFR 中提炼 `Top-10 QAS`
2. DDD 战略设计：事件风暴、聚合、不变式、上下文映射
3. 架构方案权衡：至少两套候选方案 + ATAM
4. 多视图表达：`C4` 分层图集
5. 契约与数据：OpenAPI、ER 图、数据字典、多租户与合规
6. 决策固化：ADR、12 个月架构演进路线图、PPT

主要输入：
- `lab1` 的 FR/NFR、SRS、UML、Glossary、RTM

主要产物：
- `D2-1` 质量属性场景与效用树
- `D2-2` 事件风暴与上下文映射图
- `D2-3` 候选架构对比与 ATAM 报告
- `D2-4` C4 架构图集
- `D2-5` OpenAPI 契约文档
- `D2-6` 数据模型 ER 图与数据字典
- `D2-7` ADR 集
- `D2-8` 架构演进路线图
- `D2-9` 答辩 PPT

输出给实验三的关键内容：
- 两个优先落地的核心服务边界
- API 契约
- 数据模型
- 部署与观测约束
- ADR 中的关键技术决策

### 3.4 Lab 3：构建、部署与观测

目标：
- 把实验二中选定的核心方案在 `project/` 内做成最小可运行 PoC，并具备交付能力。

预留重点：
- 优先落地 `reservation-service`、`member-service`
- 容器化与本地编排
- CI/CD 流水线
- 基础可观测性
- 部署说明与运行证据

主要输入：
- `lab2` 的 C4、OpenAPI、ER、ADR
- `project/` 代码骨架

为保证开发可控，实验三建议简化成“文档和功能优先、DevOps 能力后置”的五段式流程：

#### Phase A：先定边界与说明文档

目标：
- 先把“本轮只做什么、不做什么、如何运行”写清楚，避免工程一开始就扩散。

主要任务：
- 先更新 `project/README.md`、`docs/runbook.md`、`docs/rollback.md` 的实验三适用说明。
- 在 `docs/lab3/` 中先落 D3-1 设计方案的骨架，明确 Monorepo 取舍、双核心服务范围、最小交付边界。
- 固定本轮只优先实现 `reservation-service` 与 `member-service` 的最小闭环，不提前扩展推荐、门店运营等上下文。

完成标志：
- 同学看到文档就知道实验三的真实范围。
- 团队内部对“先做功能，后补外围”没有歧义。

#### Phase B：本地功能最小闭环

目标：
- 先把真正可演示的服务功能跑通，而不是先堆 Helm、告警和灰度。

主要任务：
- 补齐 `pyproject.toml`、`Makefile`、README 中的本地开发命令。
- 按实验二 OpenAPI 落最核心接口，至少形成一条可请求、可测试的业务链路。
- 统一服务配置、错误响应和基础日志字段。

完成标志：
- 两个服务可本地启动。
- 单测可运行。
- 至少一条业务链路可演示。

#### Phase C：容器化与本地起栈

目标：
- 在功能稳定后，再把它们装进容器，形成一键运行体验。

主要任务：
- 将 Dockerfile 升级为更规范的运行版本。
- 补齐 `docker-compose.yml`，确保 `docker compose up` 能起双服务和必要依赖。
- 增加 `make up`、`make down`、`make smoke` 之类的常用命令。

完成标志：
- 本地 compose 可启动、可停止、可冒烟验证。
- README 的本地运行链路能复现。

#### Phase D：最小 CI/CD 与部署骨架

目标：
- 只先做能交差且能解释的最小流水线，不一开始追求完整平台化。

主要任务：
- 先补 `ci.yml`，优先覆盖 `Lint -> Unit Test -> Build`。
- 再补基础镜像构建、扫描和工件输出。
- 在 `infra/helm/` 下补最小 Chart 骨架和 `dev/staging/prod` values，先满足结构完整与可讲清楚。

完成标志：
- 仓库已有最小 CI 主链。
- Helm 目录不再是空壳。
- 部署思路能在文档和答辩中说清楚。

#### Phase E：观测与交付证据收尾

目标：
- 在功能和最小流水线稳定后，再补可观测性和最终交付件，避免过早写空文档。

主要任务：
- 补结构化日志、基础 `traceId`、最小观测配置。
- 整理实验三需要的截图、扫描结果、Dashboard、DORA 表和演示脚本素材。
- 回填 D3-1 到 D3-9 中和当前实现真实对应的文档部分。

完成标志：
- 实验三既有代码，又有可验收证据。
- 文档能真实反映工程状态，而不是提前虚写。

输出给实验四的关键内容：
- 可运行服务
- 部署链路
- 监控与日志基础
- 可被测试与审查的真实代码和接口

### 3.5 Lab 4：测试、质量与 AI 审查

目标：
- 在实验三的真实工程上补齐测试体系、质量门禁与缺陷闭环。

预留重点：
- 单元 / 集成 / 端到端测试
- 契约测试与回归测试
- 质量指标与缺陷追踪
- AI 辅助代码审查记录
- RTM 中 `TC` 回填

主要输入：
- `lab1` 的 RTM 与验证结论
- `lab2` 的 OpenAPI 与 ADR
- `lab3` 的运行系统与流水线

最终输出：
- 面向课程闭环的质量工程产物
- 与实验一需求基线可追溯的测试与缺陷结果

## 4. 推荐开发顺序

如果按真实工程推进，而不是按文档顺序推进，建议遵循下面的节奏：

1. 先维护 `foundation`
2. 再冻结 `lab1` 需求基线
3. 先完成 `lab2` 的驱动力、DDD 和候选架构，再补接口、数据与 ADR
4. 用 `lab2` 的定稿结果驱动 `project/` 中的实验三最小落地
5. 最后用实验一追溯矩阵和实验二契约去约束实验四测试与缺陷闭环

### 4.1 Lab 3 简化执行顺序

建议按下面顺序推进：

1. `Phase A` 先定边界与说明文档
2. `Phase B` 本地功能最小闭环
3. `Phase C` 容器化与本地起栈
4. `Phase D` 最小 CI/CD 与部署骨架
5. `Phase E` 观测与交付证据收尾

这套顺序的核心原则是：

- 先把文档和范围写清楚，再写工程。
- 先把功能跑通，再补容器和流水线。
- 先做最小可解释版本，再补增强项。
- 交付文档尽量在真实结果出来后回填，不提前写空壳。

## 5. 当前建议

当前最适合作为主线的是：

- `lab1` 作为已完成输入基线继续沿用
- `lab2` 作为已基本定稿的架构输入继续回引
- `lab3` 作为当前工程开发主线，按模块推进
- `lab4` 先保留结构，等待实验三形成真实系统后再展开

如果后续继续扩展，优先保持下面三件事不漂移：

- 需求编号不变
- 术语口径不变
- 核心服务边界不随文档写作反复摇摆
