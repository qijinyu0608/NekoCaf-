# NekoCafé 智能预约平台 PRD

版本：v0.3
日期：2026-05-11
适用范围：`project/` 工程实现、实验三 D3-2 源代码仓库、后续实验四测试与质量闭环

## 1. 产品定位

NekoCafé 智能预约平台面向猫咪主题餐饮门店，目标不是做一个“所有角色都混在一页里”的 demo，而是逐步搭出像真实餐饮系统那样清楚分层的产品：

1. 顾客在独立预约首页完成进入、查看会员权益、查询时段、创建预约、查看我的预约。
2. 店员通过独立后台入口处理今日预约、筛选状态、确认到店与后续排台动作。
3. 运营、猫咪健康、推荐权益保留独立模块边界，不和顾客预约台混用同一套身份与状态。

当前阶段先做课程 PoC 所需的最小真实产品闭环：会话分层、顾客预约、店员后台首轮接入，以及能支持后续复杂模块继续演进的前端壳结构。

## 2. 输入依据

### 2.1 课程任务书约束

- 实验一要求形成需求基线，包含 30 条功能需求、10 条非功能需求、UML、SRS、Glossary 与 RTM。
- 实验二要求完成微服务拆分、C4、OpenAPI、ER、ADR，并优先输出桌位预约服务和会员服务接口契约。
- 实验三要求在真实工程上形成可运行代码、Docker/Compose、CI/CD、部署和可观测基础。
- 实验四要求在实验三工程上补测试、质量门禁、缺陷闭环、AI 代码审查和 RTM 扩展。

### 2.2 当前工程事实

- 仓库仍采用 `project/` monorepo。
- 后端已有 `member-service` 与 `reservation-service` 两个 FastAPI 服务。
- 前端采用 React + Vite + TypeScript，并已从单文件工作台拆成 `auth/`、`shell/`、`modules/` 三层结构。
- 本地 PoC 使用 SQLite 持久化会员、积分、时段和预约记录。
- 当前通过 `member-service` 提供 mock session 能力，不引入真实账号密码体系。

### 2.3 品牌与视觉输入

- Logo 采用 `project/frontend/src/assets/nekocafe-logo.png`。
- 品牌关键词：安静、温暖、清楚、可信赖。
- 主视觉色板来自 logo：暖白、黑灰、咖啡金、橄榄绿。
- 顾客首页应更像真实咖啡预约站，店员后台应更像工具型 console。

## 3. 用户与场景

### 3.1 顾客

目标：
- 快速进入预约首页，而不是理解后台结构。
- 知道自己当前的会员等级、积分和偏好。
- 快速找到合适门店与时段，完成预约。
- 清楚知道预约状态、何时到店、几人、主题/区域信息。
- 能在规则允许时取消自己的预约。

核心场景：
- `UC-001` 注册与维护会员资料
- `UC-002` 浏览门店与可预约时段
- `UC-003` 创建桌位预约
- `UC-004` 管理预约历史与取消
- `UC-009` 会员积分与优惠券

### 3.2 店员

目标：
- 从独立后台入口进入当日工作区。
- 查看今日预约并按状态筛选。
- 标记到店、入座、离店。
- 处理迟到、爽约、临时改台。
- 识别异常状态，减少现场沟通成本。

核心场景：
- `UC-005` 店员排台与异常处理
- `NFR-010` 店员后台支持异常状态颜色区分

### 3.3 运营与店长

目标：
- 维护门店营业日历、桌位和活动配置。
- 查看预约漏斗、复购、活动效果。
- 对敏感操作进行审计。

核心场景：
- `UC-008` 运营分析与门店配置
- `NFR-004` 新门店上线零停机
- `NFR-007` 关键运营操作保留审计日志

### 3.4 猫咪护理与门店工作人员

目标：
- 记录猫咪健康状态。
- 根据健康与互动限制影响可预约主题和推荐结果。
- 避免顾客预约到不适合互动的猫咪主题。

核心场景：
- `UC-006` 猫咪健康打卡

## 5. 身份、会话与权限策略

### 5.1 前端三层结构

- `Session/Auth Layer`
  - 管当前是谁、属于哪个租户、角色是什么、会话是否有效。
  - 顾客与店员不再依赖页面内部硬编码身份。
- `Workspace Shell`
  - 管品牌区、顶部状态、入口导航和角色对应的页面壳。
  - 顾客首页和店员后台是两个不同入口。
- `Domain Modules`
  - 顾客预约台、店员后台、猫咪健康、运营看板各自独立。

### 5.2 Mock Session 策略

- `member-service` 提供 mock session 能力。
- 顾客 persona：
  - `member-1001 / Momo / role=customer`
- 店员 persona：
  - `staff-sh-001 / Aki / role=staff / storeId=store-shanghai-001`
- 会话 token 仅服务课程 PoC，不代表正式生产鉴权方案。

### 5.3 最小权限规则

- `customer`
  - 只能访问自己的资料、积分和自己的预约。
  - 不能访问店员后台接口。
- `staff`
  - 只能访问门店后台预约列表和到店确认。
  - 不能访问顾客“我的资料/我的积分/我的预约”接口。

## 6. 信息架构

### 6.1 页面入口

- `/`
  - 顾客预约首页
  - 顾客优先，店员入口放次级位置
- `/staff`
  - 店员后台入口与店员工作台

### 6.2 顾客首页结构

1. 品牌区
   - Logo
   - 品牌主文案
   - 顾客入口说明
   - 店员后台次级入口
2. 会话与会员区
   - 继续预约入口
   - 当前会话状态
   - 会员摘要
3. 顾客工作区
   - 日期/人数选择
   - 可预约时段
   - 确认预约
   - 我的预约
4. 后续能力区
   - 猫咪健康
   - 运营看板

### 6.3 店员后台结构

1. 后台入口卡
2. 会话状态卡
3. 今日预约列表
4. 状态筛选
5. 到店确认
6. 后续后台能力入口

设计约束：
- 顾客端和店员端不共用同一套“角色切 tab”交互。
- 顾客首页不展示具体桌号，只展示时间、人数、主题/区域、状态。
- 顶部状态区只显示“会话状态 + 当前域操作状态”。
- 移动端关键预约流程不超过 3 个关键操作区。

## 7. 核心流程

### 7.1 顾客预约流程

1. 顾客打开首页。
2. 顾客以顾客身份进入预约工作区。
3. 系统恢复或创建顾客会话。
4. 系统加载会员资料和积分账户。
5. 系统按默认门店、日期、人数查询可预约时段。
6. 顾客选择日期和人数。
7. 系统刷新可预约时段。
8. 顾客选择时段并确认预约。
9. 系统创建预约记录并刷新“我的预约”。
10. 顾客可取消自己的预约。

验收标准：
- 顾客首页可独立进入，不暴露店员工具操作。
- 会员信息展示 `nickname`、脱敏手机号、等级、积分。
- 可预约时段展示开始时间、主题、容量。
- 创建预约成功后，我的预约列表出现新记录。
- 我的预约不显示具体桌号。
- 取消预约后状态变为已取消，并不再显示取消按钮。

### 7.2 店员后台流程

1. 店员进入 `/staff`。
2. 店员以店员身份进入后台。
3. 系统展示今日预约。
4. 店员按状态筛选。
5. 店员确认到店。
6. 后续版本补入座、离店、改台、迟到/爽约处理。

### 7.3 猫咪健康预留流程

1. 店员进入猫咪健康台账。
2. 选择猫咪档案。
3. 录入体温、食量、情绪、互动限制。
4. 异常指标触发提醒。
5. 健康限制影响后续可预约主题和推荐结果。

## 14. 决策记录

1. 先做 Mock Session API，而不做真实账号密码，是为了让课程 PoC 先把身份边界拆对。
2. `member-service` 承接 mock session，而不是新起 `auth-service`，是为了控制本轮范围并复用现有会员上下文。
3. 顾客首页 `Customer First`，店员后台独立进入，是为了更接近真实餐饮系统，而不是继续用同页 tab 伪装多角色。
4. 顾客侧隐藏桌号、店员侧保留桌号，是为了更贴近真实预约站与门店后台的职责边界。
5. 后续所有新功能应先补 PRD 编号，再补接口、数据表、测试和页面。
