# NekoCafé 智能预约平台 PRD

版本：v0.1
日期：2026-05-10
适用范围：`project/` 工程实现、实验三 D3-2 源代码仓库、后续实验四测试与质量闭环

## 1. 产品定位

NekoCafé 智能预约平台面向猫咪主题餐饮门店，核心价值是让顾客更可靠地完成预约，让店员更清楚地处理到店与排台，让运营方逐步获得可追溯、可观测、可扩展的门店经营系统。

本阶段不追求一次性做完完整商业平台，而是围绕课程任务书和现有工程骨架，先形成一个可运行、可演示、可测试的最小产品闭环：

1. 顾客可查看会员状态与积分权益。
2. 顾客可按门店、日期、人数查询可预约时段。
3. 顾客可创建预约、查看我的预约、取消预约。
4. 系统为店员后台、猫咪健康、运营看板、推荐权益保留清晰入口和接口边界。

## 2. 输入依据

### 2.1 课程任务书约束

- 实验一要求形成需求基线，包含 30 条功能需求、10 条非功能需求、UML、SRS、Glossary 与 RTM。
- 实验二要求完成微服务拆分、C4、OpenAPI、ER、ADR，并优先输出桌位预约服务和会员服务接口契约。
- 实验三要求从实验二中选取 2 个核心服务作为 PoC，形成可运行代码、Docker/Compose、CI/CD、部署和可观测基础。
- 实验四要求在实验三真实工程上补测试、质量门禁、缺陷闭环、AI 代码审查和 RTM 扩展。

### 2.2 现有工程事实

- 当前工程采用 `project/` monorepo。
- 后端已有 `member-service` 与 `reservation-service` 两个 FastAPI 服务。
- 前端已新增 `frontend/`，采用 React + Vite + TypeScript。
- 本地 PoC 使用 SQLite 持久化会员、积分、时段和预约记录。
- 现有 OpenAPI 中已经定义会员详情、积分账户、可预约时段、预约创建、预约详情、预约取消、会员预约列表等接口。

### 2.3 品牌与视觉输入

- Logo 采用 `project/frontend/src/assets/nekocafe-logo.png`。
- 品牌关键词：安静、温暖、清楚、可信赖。
- 主视觉色板来自 logo：暖白、黑灰、咖啡金、橄榄绿。
- UI 首屏必须是可操作工作台，不做营销型 landing page。

## 3. 用户与场景

### 3.1 顾客

目标：
- 快速知道自己是否为会员、有什么权益。
- 快速找到合适门店与时段。
- 清楚知道预约是否成功、何时到店、几人、桌位状态。
- 能在规则允许时取消预约。

核心场景：
- `UC-001` 注册与维护会员资料
- `UC-002` 浏览门店与可预约时段
- `UC-003` 创建桌位预约
- `UC-004` 管理预约历史与取消
- `UC-009` 会员积分与优惠券

### 3.2 店员

目标：
- 查看今日预约。
- 标记到店、入座、离店。
- 处理迟到、爽约、临时改台。
- 识别异常状态，减少现场沟通成本。

核心场景：
- `UC-005` 店员排台与异常处理
- `NFR-010` 店员后台支持异常状态颜色区分

### 3.3 运营与店长

目标：
- 维护门店营业日历、桌位和活动配置。
- 查看预约漏斗、复购、活动效果。
- 对敏感操作进行审计。

核心场景：
- `UC-008` 运营分析与门店配置
- `NFR-004` 新门店上线零停机
- `NFR-007` 关键运营操作保留审计日志

### 3.4 猫咪护理与门店工作人员

目标：
- 记录猫咪健康状态。
- 根据健康与互动限制影响可预约主题和推荐结果。
- 避免顾客预约到不适合互动的猫咪主题。

核心场景：
- `UC-006` 猫咪健康打卡

V1 页面采用平台型布局：

1. 左侧品牌与导航区

设计约束：

- 顾客预约台必须是首屏主要操作，不隐藏在说明文案之后。
- 移动端关键预约流程不超过 3 个关键页面或操作区。
- 复杂后台能力先展示入口和能力卡片，不在 V1 制造不可用的假功能。
- 所有后续模块入口必须能映射到实验一需求和实验二限界上下文。

## 6. 核心流程

### 6.1 顾客预约流程

1. 系统加载会员资料和积分账户。
2. 系统按默认门店、日期、人数查询可预约时段。
3. 顾客选择日期和人数。
4. 系统刷新可预约时段。
5. 顾客选择时段。
6. 顾客点击确认预约。
7. 系统创建预约记录。
8. 系统刷新我的预约列表。
9. 顾客可取消已预约记录。

验收标准：

- 会员信息展示 `nickname`、脱敏手机号、等级、积分。
- 可预约时段展示开始时间、主题、容量。
- 创建预约成功后，我的预约列表出现新记录。
- 我的预约显示预约状态、门店、人数、桌位。
- 取消预约后状态变为已取消，并不再显示取消按钮。

### 6.2 店员后台预留流程

V1 仅保留入口，V2 实现：

1. 店员进入店员后台。
2. 系统展示今日预约。
3. 店员按状态筛选。
4. 店员确认到店并分配桌位。
5. 店员处理迟到、爽约、取消和临时改台。

### 6.3 猫咪健康预留流程

V1 仅保留入口，V3 实现：

1. 店员进入猫咪健康台账。
2. 选择猫咪档案。
3. 录入体温、食量、情绪、互动限制。
4. 异常指标触发提醒。
5. 健康限制影响后续可预约主题和推荐结果。

## 10. UI 与交互要求

1. 使用真实 logo：`project/frontend/src/assets/nekocafe-logo.png`。
2. 卡片半径不超过 8px，整体保持工具型界面，不做过度装饰。
3. 按钮使用图标加文字，图标优先使用 `lucide-react`。
4. 文案要表达业务状态，不写“这里是某功能说明”一类占位废话。
5. 移动端不允许按钮文字溢出或模块互相遮挡。
6. 后续能力可以是卡片入口，但要明确“预留”或“后续后台能力”，避免误导为已实现。

## 13. 决策记录

1. V1 先做顾客预约台，因为它同时覆盖会员服务和预约服务，是实验三最小可运行 PoC 的主链路。
2. V1 使用 SQLite，因为课程本地演示和测试更稳定；长期生产型数据库仍沿用实验二 ADR 中的 MySQL 方向。
3. 前端先做平台信息架构，不只做单页 demo，是为了给店员后台、猫咪健康、运营看板和实验四测试留出扩展位置。
4. 后续所有新功能应先补 PRD 编号，再补接口、数据表、测试和页面。
